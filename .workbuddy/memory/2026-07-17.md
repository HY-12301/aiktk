## gist 同步无法同步 — 诊断与修复（方案B）

- 现象：用户反馈 gist 无法同步。诊断：右上角云朵 badge 显示「离线/仅本机」= 浏览器 `fetch api.github.com` 失败，卡在网络层。
- 实测：自身环境直连 `api.github.com` 返回 HTTP 200（GitHub 服务正常）。根因=用户浏览器所在网络直连 api.github.com 被墙/超时（题库走 jsdelivr 有国内加速不受影响，故题库能加载、同步不行）。用户本机无任何代理软件。
- 方案：用户选「方案B 加 Worker 中继」。
- 改动（index.html，commit 318230a，已直连推送上线）：
  - `SYNC_API` 常量 → `syncApiBase()` 函数，读 `localStorage.syncProxy`；非空则基址= `proxy/gists`，否则直连 `https://api.github.com/gists`。4 处 fetch 引用全部改完。
  - `syncHeaders()` 兼容 fine-grained token：`SYNC_TOKEN` 以 `github_pat_` 开头用 `Bearer` 前缀，否则沿用 `token` 前缀（原硬编码只支持 classic PAT，fine-grained 会 401）。
  - `openSyncSettings` 新增「同步代理地址」输入框 + 说明；`saveSyncToken` 保存 `syncProxy`。
- 新增 `cloudflare-sync-worker.js`：纯转发 Worker（透传 Authorization + 加 CORS 头 + OPTIONS 预检直回），用于国内绕开 api.github.com 封锁。用户需在 Cloudflare 新建 Worker 粘贴部署，再把地址填进「同步代理地址」。
- 安全说明：Worker 纯转发，安全模型与原“PAT 存浏览器”一致，未额外放大风险。
