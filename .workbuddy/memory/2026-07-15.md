## 加载提速分析（本回合补充，已精确复测）

- 下载量 8.69MB 已被 SW 缓存，二次访问不重复下载；手机瓶颈是 CPU 解析 40MB，非网速。

## 卡顿根因2：点「开始刷题」也卡（已修复并部署）
- 症状：首次加载完成后，之后每次点「开始刷题」仍卡好几秒。
- 路径：数据已加载(D=true)时 startQuiz 直接走 doS()（index.html:946），不重新下载，但 doS 出题调度极慢。
- 根因：`doS` 对 1.7 万题逐题调用 `geM()`/`topicKey()`/`kwWeakness()` 判断 bucket，而三者每次都走 `GS()`→`LS()`→`JSON.parse(localStorage.getItem('s'))`，即把整个进度状态重复解析。调用量：过滤~1.7万(geM) + 分类~5万(geM+topicKey+kwWeakness 各1.7万) + 排序(尤其 weak 桶每比较多次调 topicKey/kwWeakness) → 单次点击 JSON.parse 进度状态 十万~几十万次。手机 CPU 慢→卡几秒。
- 与「启动卡」区分：启动卡=解析40MB题库；本次卡=重复解析进度状态，本质都是「每次重新 parse」。
- 修法（极小）：缓存解析结果到模块级变量——`GS()` 首次才 parse、之后返回缓存；`SS(d)` 写入同步缓存；import 处(2333附近)刷新缓存。改动 3~4 行，doS 从几万次 parse 变几次对象读取。
- 状态：用户说「先改这个」（提速方案 A/B 先不动）。已实施并部署 Cloud Studio（同 sandboxId e2fd3948585f431bb2122930ed5363fd）：GS() 改为缓存解析结果（STATE 模块级变量，首次 parse、之后返回缓存；SS(d) 同步 STATE；import 走 SS 保证缓存一致）。同步路径 pullGist→SS / cloudSyncPush→LS 均兼容，无失效。node --check 语法通过。提速方案 A/B 仍按用户要求暂未做。

## 方案1 IndexedDB 缓存 崩溃风险检查（16:0x）
- 代码事实：题库全局变量是 `Q`（`var Q=[]` 704行），`D` 只是"加载完"布尔；`loadData(cb,errCb)` 是回调式异步，业务全在 `cb` 内（`Q=d;D=true;doS()` 950 / 1844 / 1964）；`Q` 来自 `JSON.parse(txt)`，纯数据数组（无方法/类实例）。
- 现状隐藏卡点：`loadData` 缓存命中走 `r.json().then(cb)`（895行），`r=new Response(txt)` 仍再 JSON.parse 40MB → 每次都重解析（SW 缓存只省下载+解压）。
- 结论：方案1 **不会结构性崩溃**——Q 是纯数据，IndexedDB 结构化克隆保真；loadData 本就异步，业务在 cb 内，初始化顺序不变。
- 三大实现风险（须防护）：① IndexedDB 不可用/配额满（隐私模式、WebView 禁用、QuotaExceededError）未回退 → loadData 失败白屏；② 失效机制缺失 → 题库升级后旧 IDB 对象缺新字段 → 字段 undefined 崩溃；③ IDB 与 SW `ai-questions` 缓存失效不同步 → 数据不一致。
- 防护清单（实施时必做）：IDB 优先+命中即 cb 并返回（避免双 cb）；IDB 失败/配额满 catch 后回退原 fetch+parse（走 errCb 而非静默挂起）；用题库版本号(hash/构建号)作为 IDB store key 或校验值，版本变则清旧重建；IDB 与 SW 缓存共用同一版本号校验。
- **已实施（16:1x）**：按防护清单全做。index.html `loadData` 改为 **IDB 优先**（`idbGet` 命中即 `cb(cached)` 返回；miss 走原 fetch+parse+`idbPut`）；新增 `idbOpen/idbGet/idbPut`（DB `aiktk`/store `qbank`，`indexedDB` 不可用/配额满均 catch 回退原逻辑，绝不静默挂起）。版本失效：`BANK_VER='1'`，`IDB_KEY='qbank-'+BANK_VER`，`DATA_CACHE='ai-questions-'+BANK_VER`；`sw.js` 的 `KEEP` 同步改为 `'ai-questions-'+BANK_VER`。**更新题库时必须同步 bump 两处 `BANK_VER`**（index.html + sw.js）。`node --check` 语法通过。已部署 Cloud Studio（同 sandboxId）。用户需硬刷新清旧 SW cache（旧 `ai-questions` 会被新 SW activate 清理，旧 IDB key `qbank-旧` 不再被读取）。未动题库/业务逻辑，刷题系统不受影响。
