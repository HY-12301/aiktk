# 新算法适配分析报告

## 一、新增的数据与函数

新算法引入以下内容，均已实现：

| 新增项 | 位置 | 作用 |
|--------|------|------|
| `d.mem` 字段 | localStorage `s` 键 | 存储每道题的记忆档案（ef/interval/phase/reps/nextReview 等） |
| `geM(qid)` | 全局函数 | 读取单题记忆状态 |
| `seM(qid, state)` | 全局函数 | 写入单题记忆状态 |
| `mkM()` | 全局函数 | 创建默认记忆档案 |
| `updateMemState(q, ok)` | 全局函数 | SM-2 算法核，答题后更新记忆状态 |
| `doS()` 重写 | 出题函数 | 使用记忆优先级排序替代旧的 done 排序 |

---

## 二、需调整的页面/功能（按优先级排列）

### 问题1：云端同步未包含 mem 字段 ⚠️ 高优先级

**位置**：`cloudSyncPull()`（第547行）

**现状**：拉取时不合并 `mem`，因此换设备后所有记忆档案丢失，又得从零开始。

**需调整**：
```javascript
// 第557行附近，新增一行：
m.mem = mergeObj(local.mem||{}, remote.mem||{});
```

---

### 问题2：智能出题（单科）仍用旧 done 排序 ⚠️ 中优先级

**位置**：`doSmartExam()` 单科路径（第1530-1541行）

**现状**：判断"未做过"用的是 `GS().done`（旧的完成记录），没用 `geM()` 获取记忆优先级。出题顺序仍是随机的，没有把快遗忘的题提前。

**需调整**：将单科出题的题目池排序改为与 `doS()` 一致的记忆优先级排序。

---

### 问题3：智能组卷（卷一/卷二）仍用旧 done 排序 ⚠️ 中优先级

**位置**：`doSmartExam()` 组卷路径（第1588-1594行）

**现状**：同问题2，每个题型子池的排序仍基于旧的 `GS().done`。

**需调整**：改为用 `geM()` 做记忆优先级排序。

---

### 问题4：错题刷题未使用记忆优先级 ◇ 低优先级

**位置**：`doQuizFromErrors()`（第1382-1392行）

**现状**：从错题本刷题时，直接 `for` 循环随机打散，没有按"最该复习的错题优先"排序。

**需调整**：错题也应该优先出那些快遗忘的（记忆档案中 phase 为 relearning 或 overdue 的），确保薄弱点被重点强化。

---

### 问题5：统计页面未展示记忆数据 ◇ 低优先级

**位置**：`rP()`（进度/统计页）

**现状**：统计页显示做题数、正确率、完成进度，但没有显示"需复习题数""已掌握题数"等基于记忆状态的数据。

**建议**：新增两张指标卡片：
- "待复习"：`mem.nextReview <= 今天` 的题目数
- "已掌握"：`ef >= 2.5 且 interval >= 90` 的题目数

---

### 问题6：刷题页面无记忆状态提示 ◇ 低优先级

**位置**：`qR()`（题目渲染）

**现状**：用户看不到当前这道题的记忆状态（是新题？是复习？是第几次出现？）。

**建议**：在题目类型标签旁边加一个小标签：🆕新题 / 🔄复习(第3次) / ⚠快忘了

---

### 问题7：清空错题本未清理关联记忆档案 ◇ 低优先级

**位置**：`clearErrors()`（第1247行）

**现状**：只清除了 `d.w`，对应题目的 `mem` 档案仍在（phase 可能还是 learning 或 relearning）。

**需调整**：清空错题本时同步重置对应题目的 memory 状态到初始值。

## 三、无需调整的功能（已验证兼容）

| 功能 | 原因 |
|------|------|
| `submitAnswer()` | 已调用 `updateMemState()`，评分和间隔更新正常工作 |
| `rW()`（记录错题） | 只操作 `d.w` 和 `d.done`，与 `mem` 独立，不冲突 |
| `eQ()`（总结页） | 使用 `WX`（会话级数据），不依赖 `mem` |
| `endExam()` | 走正常 `submitAnswer` 流程，每题自动更新 mem |
| `GS()` / `SS()` | 已初始化 `d.mem || {}` |
| `rEB()`（错题本） | 只读 `d.w`，与 `mem` 独立 |

## 四、性能影响评估

| 维度 | 影响 |
|------|------|
| localStorage 读 | 每次调用 `geM()` 触发一次 `JSON.parse`（GS 读取全量），小量数据无影响 |
| localStorage 写 | 每次 `submitAnswer` 触发一次 `JSON.stringify`（seM 写入全量），约 50KB，可接受 |
| 出题排序 | `doS()` 中 `pool.forEach` + `geM` 查 O(n) 遍历，17k 题目约 5ms，可接受 |
| 内存占用 | 每题新增约 100 字节的 mem 数据，17k 题约 1.7MB，在 localStorage 5MB 限制内 |
| 云同步 | 新增字段会导致同步数据量增加约 1.7MB，可能超过 jsonblob 免费限额需关注 |

---

## 五、建议执行顺序

```
1. 修复云端同步的 mem 合并 [高]
2. 将智能出题（单科+组卷）改为记忆优先级排序 [中]
3. 错题刷题加入记忆优先级 [低]
4. 统计页增加复习/掌握指标 [低]
5. 刷题页增加记忆状态标签 [低]
6. 清空错题关联清理 [低]
```

## 六、存在的一个潜在 Bug（已在代码中）

**位置**：`submitAnswer()` → `updateMemState(q, ok)` 中的 `geM(q.id)` 调用

**问题**：`geM` 内部调用 `GS()`，而 `submitAnswer` 之前已经调用了 `GS()` 并修改了 `d.done` 和 `d.w`，然后再 `SS(d)`。之后 `updateMemState` 中 `geM` 又会重新 `GS()` 读取完整数据。由于 `submitAnswer` 中的 `SS(d)` 已经将 done/w 写入 localStorage，`geM` 重新读能拿到最新值。流程正确，但多了一次不必要的读写。

**优化建议**：将 `d` 引用传入 `updateMemState`，避免重复 `GS()` 调用。
